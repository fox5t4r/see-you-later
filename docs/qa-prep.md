# 발표 Q&A 대비 — 의사결정 과정 정리

이 문서는 발표 시 예상되는 질문과 답변을 정리합니다.
핵심은 **"왜 이렇게 했는가"** — 기술적 선택의 근거와 의사결정 과정입니다.

---

## Q1. 이 프로젝트를 왜 만들었나요?

**핵심 답변**: 콘텐츠 과잉 시대에 "읽을 시간이 없는 문제"를 해결하고 싶었습니다.

**상세**:
- 매일 수십 개의 기사, 유튜브 영상, 논문이 쌓이지만 다 볼 시간이 없음
- 기존 요약 도구(TLDR This, Glasp 등)는 단순 요약만 제공 — "이걸 읽을 가치가 있는지"를 판단해주지 않음
- **차별화 포인트**: 단순 요약이 아닌 "추천도 평가"로 읽을 가치를 판단 + "Watch Later 자동 요약"으로 유튜브 영상까지 커버
- 특히 YouTube "나중에 볼 동영상"에 영상을 추가만 하고 실제로 보지 않는 문제를 해결하고 싶었음

---

## Q2. 왜 크롬 확장 프로그램으로 만들었나요?

**핵심 답변**: "지금 보고 있는 페이지를 바로 요약"하려면 브라우저 확장이 유일한 선택지였습니다.

**상세**:
- 웹 앱으로 만들면 사용자가 URL을 복사 → 붙여넣기 → 요약 요청해야 함 (3단계)
- 크롬 확장은 현재 탭의 콘텐츠에 직접 접근 가능 → 버튼 한 번으로 요약 (1단계)
- 이 "한 단계 줄이기"가 사용자 경험에서 결정적 차이를 만듦
- 또한 Service Worker가 백그라운드에서 동작하므로 Watch Later 자동 요약 같은 기능이 가능

---

## Q3. 왜 Gemini를 선택했나요? GPT나 Claude가 아니라?

**핵심 답변**: 무료 티어 + YouTube URL 직접 처리가 가능한 유일한 모델이었습니다.

**상세**:
- 처음에는 Claude Haiku를 사용했음 (v0.1.0)
- 문제: Claude API는 무료 티어가 없어서 사용자가 즉시 비용을 부담해야 함
- GPT도 마찬가지로 무료 티어 없음
- Gemini는 하루 약 250회 무료(Gemini 3 Flash Preview 기준) + YouTube URL을 직접 전달하면 영상을 분석해주는 멀티모달 기능 제공
- 이 조합은 "사용자 비용 $0"과 "자막 없는 영상도 요약 가능"을 동시에 달성할 수 있는 유일한 선택지

**보충 (비교표)**:
| | Claude | GPT | Gemini 3 Flash Preview |
|---|---|---|---|
| 무료 티어 | ❌ | ❌ | ✅ (하루 ~250회) |
| YouTube 멀티모달 | ❌ | ❌ | ✅ (URL 직접 전달) |
| 요약 품질 | 우수 | 우수 | 우수 |
| 사용자 비용 | 유료 | 유료 | **무료** |

---

## Q4. 처음부터 Gemini를 쓰지 않고 왜 Claude로 시작했나요?

**핵심 답변**: 프로젝트 시작 시점에는 Gemini의 무료 티어와 멀티모달 기능이 지금만큼 안정적이지 않았고, Claude Haiku가 가성비 최고였습니다.

**상세**:
- v0.1.0 개발 시점에 Claude Haiku 4.5가 속도 대비 품질이 가장 좋았음
- 이후 Gemini 2.5가 무료 티어를 대폭 확대하고 YouTube URL 직접 처리를 지원하면서 전환 결정
- 이 전환 과정 자체가 "기술 환경 변화에 대응하는 의사결정"의 좋은 사례
- Docker 백엔드까지 제거하면서 설치 난이도를 획기적으로 낮춤

---

## Q5. 왜 SDK 대신 REST API를 직접 호출하나요?

**핵심 답변**: 크롬 확장의 Service Worker 환경에서 SDK 호환성이 보장되지 않았기 때문입니다.

**상세**:
- `@google/generative-ai` (공식 SDK v1) → 2025년 말 deprecated
- `@google/genai` (공식 SDK v2) → Node.js 전용, 브라우저 Service Worker에서 동작 보장 안 됨
- Service Worker는 일반 브라우저 환경과 다름 — DOM 없음, window 없음, 생명주기 제한
- REST API는 `fetch()` 하나로 동작 → 어떤 환경에서든 100% 동작 보장
- SDK 버전 업데이트나 deprecation에 영향받지 않는 장점

---

## Q6. YouTube 자막은 어떻게 가져오나요?

**핵심 답변**: YouTube의 비공식 Innertube API를 역공학하여 자막을 추출합니다.

**상세**:
- YouTube Data API v3를 사용하면 OAuth 인증이 필요 → 사용자에게 Google 로그인 동의 화면 표시 → UX 저하
- 대신 YouTube 내부적으로 사용하는 Innertube API를 분석하여 자막 데이터를 직접 가져옴
- 이 방식은 인증 없이 자막을 가져올 수 있어 사용자 경험이 훨씬 좋음
- 자막이 없는 영상은 Gemini 멀티모달로 YouTube URL을 직접 전달하여 영상 자체를 분석

**하이브리드 전략의 이유**:
- 자막 있는 영상 (~80%): 텍스트만 보내면 토큰 비용 1/10 → 무료 한도 절약
- 자막 없는 영상 (~20%): 멀티모달로 영상 자체 분석 → 비용은 높지만 처리 가능
- 무조건 멀티모달로 보내면 무료 한도를 하루 만에 소진할 수 있음

## Q7. Watch Later 목록은 어떻게 가져오나요?

**핵심 답변**: YouTube 페이지의 초기 데이터(`ytInitialData`)를 파싱하여 추출합니다.

**상세**:
- 처음에는 YouTube Data API v3를 고려 → OAuth 필요 → 포기
- 두 번째로 YouTube 페이지 DOM을 파싱 시도 → YouTube가 SPA라서 DOM이 동적 변경 → 불안정
- 최종 해결: YouTube 페이지 로드 시 `window.ytInitialData`에 JSON 형태로 초기 데이터가 존재
- `chrome.scripting.executeScript`로 YouTube 탭에 스크립트를 주입하여 이 데이터를 추출
- 이 방식은 DOM 변경에 영향받지 않아 가장 안정적

**3번의 전면 재구현을 거친 이유**:
1. 1차: Service Worker에서 직접 fetch → CORS 문제
2. 2차: Content Script로 DOM 파싱 → YouTube SPA 특성상 불안정
3. 3차: `chrome.scripting` + `ytInitialData` → 안정적 동작

## Q8. AI 도구를 어떻게 활용했나요? (바이브 코딩 관련)

**핵심 답변**: AI 도구마다 강점이 다르기 때문에 역할을 분담했습니다. 중요한 것은 "AI에게 무엇을 시킬지 결정하는 것"이 사람의 역할이라는 점입니다.

**상세**:
```
Gemini + ChatGPT → 아이디어 탐색, 시장 조사
Claude Opus      → 요구사항 명세, 아키텍처 설계, 트러블슈팅, 문서화
Claude Sonnet    → 코드 생성, 단순 버그 수정
Gemini 3.1       → 프론트엔드 UI/UX 디자인
```

- **바이브 코딩이지만 의사결정은 사람이 함**: AI가 코드를 생성하지만, "무엇을 만들지", "어떤 기술을 쓸지", "어떤 문제를 먼저 풀지"는 모두 사람이 결정
- **AI 도구 선택도 의사결정**: Opus는 추론 능력이 뛰어나서 설계/트러블슈팅에, Sonnet은 속도가 빨라서 구현에, Gemini는 시각적 감각이 좋아서 UI에 배치
- **프롬프트 품질이 결과 품질을 결정**: "요약해줘"보다 "JSON 형식으로, 3줄 요약 + 추천도 1~5점 + 추천 대상을 포함해서 요약해줘"가 훨씬 좋은 결과

## Q9. 가장 어려웠던 부분은 무엇인가요?

**핵심 답변**: Watch Later 목록 추출과 Gemini 응답 파싱이 가장 어려웠습니다.

**상세**:

**Watch Later 추출 (3번 재구현)**:
- YouTube는 SPA(Single Page Application)라서 일반적인 웹 스크래핑 방식이 통하지 않음
- 페이지 로드 타이밍, DOM 구조 변경, CORS 정책 등 여러 장벽
- 최종적으로 `ytInitialData` 파싱이라는 안정적인 방법을 찾기까지 3번의 시행착오

**Gemini 응답 파싱**:
- Gemini 2.5 Flash에서 "thinking" 파트가 응답에 포함되는 문제 발견
- JSON만 기대했는데 thinking 텍스트 + JSON이 섞여서 파싱 실패
- thinking 파트를 필터링하는 로직을 추가하여 임시 해결
- v0.2.11에서 Gemini 3 Flash Preview로 전환 + `thinkingBudget=0` 설정으로 근본 해결

## Q10. v0.1.0에서 Docker 백엔드를 왜 만들었다가 삭제했나요?

**핵심 답변**: 처음에는 Whisper 음성 변환을 위해 필요했지만, Gemini가 YouTube URL을 직접 처리할 수 있게 되면서 불필요해졌습니다.

**상세**:
- v0.1.0 아키텍처: YouTube 영상 → yt-dlp로 오디오 추출 → Whisper로 텍스트 변환 → Claude로 요약
- 이 파이프라인은 FastAPI 서버 + Docker가 필요
- 문제: 비전공자에게 Docker 설치는 사실상 불가능한 진입 장벽
- Gemini 2.5가 YouTube URL을 fileData로 직접 받아서 영상을 분석할 수 있게 되면서 전체 파이프라인이 불필요해짐
- 결과: 서버 없이 크롬 확장만으로 모든 기능 동작 → 설치 난이도 획기적 감소

---

## Q11. 추천도 평가는 어떻게 동작하나요? 신뢰할 수 있나요?

**핵심 답변**: AI에게 까다로운 평가 기준을 프롬프트로 주입하여 대부분 2~3점이 나오도록 설계했습니다.

**상세**:
- 프롬프트에 "대부분의 콘텐츠는 2~3점이 적절합니다"라고 명시
- 5점은 "상위 5%, 해당 분야 필독" 수준으로 매우 엄격하게 설정
- 추천 대상과 스킵 대상을 함께 제공하여 사용자가 자신에게 맞는지 판단 가능
- "요약으로 충분한지, 원본을 봐야 하는지"까지 안내하여 시간 투자 의사결정 지원
- 100% 정확하지는 않지만, "이 콘텐츠에 시간을 투자할 가치가 있는가"의 1차 필터로 충분

## Q12. 이 프로젝트의 한계는 무엇인가요?

**솔직한 답변**:

1. **크롬 전용**: Firefox, Safari 등 다른 브라우저 미지원
2. **Gemini 무료 한도 의존**: 하루 ~100회 제한, 초과 시 유료 전환 필요
3. **YouTube 비공식 API 의존**: Innertube API나 ytInitialData 구조가 변경되면 업데이트 필요
4. **오프라인 불가**: 인터넷 연결과 Gemini API가 필수
5. **긴 콘텐츠 품질**: 매우 긴 문서(10,000자+)는 요약 품질이 다소 떨어질 수 있음

**향후 개선 방향**:
- Chrome Web Store 정식 배포
- 요약 품질 피드백 시스템 (사용자가 요약 품질을 평가)
- 팀 공유 기능 (여러 사람이 같은 요약을 공유)
- PDF 직접 업로드 요약
